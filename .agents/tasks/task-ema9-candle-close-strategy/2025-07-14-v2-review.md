# EMA 9 candle-close strategy with multi-indicator choppy filter (v2)

Three of the five v1 issues were fixed: ATR ratio and variance ratio voters now receive computed values, EA swing levels serve as fallback when the Python bar buffer is insufficient, and breakeven MODIFY_SL only sets `_be_moved` after a successful write. The remaining two issues (single-minute race window and reversal cold start) are documented as acceptable limitations with inline comments.

**Watch for:** The variance ratio voter remains dead — `REGIME_VARIANCE_RATIO_LOOKBACK = 20` requires 21 bars, but `_completed_bars` has `maxlen=20`, so the `len(completed_bars) >= vr_lookback + 1` guard can never pass (confirmed). The EA's `InpBreakevenProfit` defaults to `$0.30`, meaning the EA moves SL to entry + $1 after just $0.30 profit — contradicting the stated $5 threshold that Python enforces (confirmed). The filter is 4-of-5 voters at best, not 5-of-5.

## High-level view

The ATR ratio voter fix works: `compute_atr` needs 15 bars, the deque holds 20, so the voter activates after ~15 minutes. The variance ratio voter has a different constraint: it needs `REGIME_VARIANCE_RATIO_LOOKBACK + 1 = 21` bars, exceeding the deque's 20-element capacity. The filter operates as a 4-indicator system at steady state, not 5.

`compute_swing_sl` receives `ea_swing_high` and `ea_swing_low` and uses them when `completed_bars` has fewer than 5 bars. Since the EA reads 20 bars of MetaTrader history immediately, the first trade after startup gets a meaningful SL rather than the $0.50 minimum fallback.

The breakeven retry logic is correct from Python's side, but the EA independently runs its own breakeven check on every tick at `InpBreakevenProfit = $0.30` — far below Python's `$5.00`. The EA will move the SL to entry + $1 within seconds of entry on any typical XAUUSD move, making Python's $5 threshold unreachable.

## Variance ratio: off-by-one against deque capacity

The computation in `main.py` lines 246-255:

```python
vr_lookback = Config.REGIME_VARIANCE_RATIO_LOOKBACK  # 20
if len(completed_bars) >= vr_lookback + 1:  # needs >= 21
    close_arr = bars_df["Close"].values[-(vr_lookback + 1):]
```

`tick_collector._completed_bars` is a `deque(maxlen=20)`. It can never hold 21 elements. The `computed_variance_ratio` stays at its default of `1.0`, and the choppy filter's VR check (`variance_ratio > 0 and variance_ratio < 0.5`) never triggers because `1.0 >= 0.5`.

Fix: `REGIME_VARIANCE_RATIO_LOOKBACK = 19` (so the check needs `>= 20`, achievable with the existing deque). This doesn't affect memory or any other consumer of the deque.

## EA breakeven fires before Python's threshold

The EA's `ManageOpenPositions()` runs on every tick:

```c
double beProfit = (g_brain_be_profit > 0) ? g_brain_be_profit : InpBreakevenProfit;
// g_brain_be_profit = 0 (default), InpBreakevenProfit = 0.30
double beLock = 1.00;
if(profit >= beProfit)  // profit >= $0.30
    newSL = entryPrice + beLock;
```

On XAUUSD, a $0.30 move happens within seconds on most entries. The EA moves SL to entry + $1 almost immediately. Python's `_check_breakeven` at $5.00 becomes permanently redundant — the SL is already at breakeven before $5 profit is ever achieved.

The user instruction says "$5 profit -> move SL to $1 profit." The EA input should be `InpBreakevenProfit = 5.00` to match. The comment in the code says "// Brain-controlled threshold or default $5/$1" but the actual input value contradicts this.

## Warmup filter sensitivity

During the first 15 minutes:
- ADX, Choppiness Index, BB squeeze: active immediately (from EA pipe)
- ATR ratio: inactive (needs 15 bars)
- Variance ratio: always inactive (deque too small)

With `RANGING_FILTER_AGREEMENT = 2`, any two of ADX/CI/BB blocking means no trade from tick one. In a trending session that happens to have low BB width (common during Asian session opens), the filter blocks entries until BB width widens — potentially missing the first 15-20 minutes of a trend. The "5-indicator" design is effectively 3 indicators during session opens.
