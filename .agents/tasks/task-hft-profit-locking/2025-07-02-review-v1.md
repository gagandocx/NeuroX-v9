# Three-phase profit-locking intelligence for HFT gold scalper

This change adds a three-layer decision system on top of the existing momentum scalper: instant reversal detection and profit locking (Phase 1), predictive intelligence via price acceleration and micro-patterns (Phase 2), and self-optimizing parameter tuning with Kelly criterion sizing (Phase 3). The implementation introduces three new modules (`intelligence.py`, `performance.py`, `optimizer.py`), extends `tick_collector.py` with velocity/exhaustion detection, extends `bridge.py` with brain settings and performance persistence, and wires everything through the main loop. The approach is additive — the existing filter chain (cooldown, direction lock, hysteresis, persistence, ATR, tick consistency) remains intact, with new signals and adjustments layered on top.

Watch for: auto-tune mutates class-level Config attributes at runtime, creating invisible coupling between trade history and threshold behavior across all code paths (confirmed). Scalp exit tracking uses a single global slot, so only the most recent signal is tracked for quick-profit exit — earlier concurrent positions are silently dropped from scalp monitoring (confirmed). The V-reversal pattern fires directly through `fire_signal` bypassing the weighted-direction disagreement check that gates all other signal paths (confirmed).

## High-level view

The tick velocity spike and scalp exit form a fast-path that wraps around the existing loop. Velocity spikes bypass the persistence filter (intentional — speed over confirmation), and the scalp exit fires `CLOSE_FULL` every tick if profit conditions are met. Both respect cooldown and direction lock but skip the other filters.

Predictive intelligence (acceleration, weighted direction, micro-patterns, adaptive thresholds) acts as additional gating and confirmation. The weighted tick direction serves as a veto — if it disagrees with momentum, the signal is skipped. Micro-patterns inject new signals (V-reversals) or tighten trailing (double-tops, rejection wicks). Adaptive thresholds scale the velocity/scalp/momentum numbers by ATR ratio, but they only affect the bar-based path — the tick-only path still uses raw Config values, creating a behavioral asymmetry between pre-bar and post-bar trading.

Self-optimization mutates `Config.MOMENTUM_THRESHOLD`, `Config.TICK_MOMENTUM_THRESHOLD`, and `Config.COOLDOWN_SECONDS` in-memory every 10 trades when win rate deviates from the 40-60% band. This is a global side-effect: all filter checks, capacity gates, and signal thresholds see the mutated values without any indication that they've changed from their initial settings. There's no decay or reset mechanism, so a hot streak followed by regime change can leave thresholds permanently tightened.

Position sizing combines three independent multipliers — streak-based lot scaling, Kelly criterion fraction, and time-of-day aggression — which stack multiplicatively. The max lot cap (0.05) prevents blowout, but the interaction between Kelly (which wants to go large after a good run) and time-profiler aggression (up to 2x) means effective sizing can jump discontinuously when thresholds align.

## Velocity spike and scalp exit: the fast-path layer

The scalp exit runs every tick (`check_scalp_exit`) and fires `CLOSE_FULL` when unrealized profit hits $0.30 within 10 seconds of entry. The tracking is a single global slot (`scalp_entry_price`, `scalp_entry_time`, `scalp_direction`), which means each new `fire_signal` call overwrites the previous scalp state. With `MAX_POSITIONS=5`:

```
Signal 1 (BUY @ 2000) -> scalp tracks 2000
Signal 2 (BUY @ 2001) -> scalp now tracks 2001, signal 1 is lost from scalp monitoring
```

Positions 1-4 can never trigger a scalp exit once position 5 fires. Those positions still benefit from EA-side trailing (via brain_settings), but the Python-side quick-profit exit is effectively single-position.

## Auto-tune and threshold mutation

`_apply_auto_tune` modifies `Config.MOMENTUM_THRESHOLD`, `Config.TICK_MOMENTUM_THRESHOLD`, and `Config.COOLDOWN_SECONDS` directly on the class. These are the same attributes read by `compute_momentum`, `has_capacity` (which uses `TICK_MOMENTUM_THRESHOLD * STRONG_MOMENTUM_MULT`), and `can_trade`. The mutation happens every 10 trades.

The feedback loop: high win rate -> thresholds tighten 10% -> more signals fire -> more trades -> if they keep winning, thresholds tighten again. There's no floor defined in `suggest_parameters` (it just multiplies by 0.90 repeatedly). After 50 consecutive wins, `MOMENTUM_THRESHOLD` would be `0.60 * 0.90^5 = 0.354`, which is substantially more aggressive than designed. The widening direction (1.20x on losses) is asymmetric — it recovers faster — but there's no absolute bound preventing thresholds from compressing below meaningful levels.

## V-reversal bypasses the weighted-direction veto

V-reversal signals from `detect_micro_patterns` fire through `fire_signal` directly in the main loop:

```python
for pat in micro_patterns:
    if pat["pattern"] == "v_reversal":
        rev_dir = pat["direction"]
        if can_trade() and direction_matches_open(rev_dir):
            fire_signal(bridge, rev_dir, current_price, "V-REVERSAL", ...)
```

A V-reversal BUY can fire while weighted direction is firmly SELL. V-reversals are by definition counter-trend, so the weighted direction *will* disagree — but this means the veto is inconsistently applied: it blocks momentum signals but never pattern signals. If the intent is that patterns override the veto, that logic deserves a comment; if it's accidental, it's a gap.

## Adaptive thresholds and the tick-only asymmetry

`compute_adaptive_thresholds(current_atr, avg_atr)` scales velocity ($0.30), scalp ($0.50), and momentum ($0.60) thresholds by the ATR ratio. The velocity spike detector and scalp exit always read directly from `Config.TICK_VELOCITY_THRESHOLD` and `Config.SCALP_PROFIT_THRESHOLD`, which are never modified by the adaptive system.

This means the adaptive threshold computation is effectively dead code for the fast-path features. The velocity spike and scalp exit always operate at fixed $0.30 thresholds regardless of current volatility. During high-volatility periods (London/NY open), $0.30 moves are noise, not spikes — the system will fire velocity spikes on what is actually normal price action.

## Pattern detection: double-top false positive risk

Double-top detection searches for peaks separated by at least 3 ticks within $0.10 tolerance. In a 30-tick window at HFT speeds, local peaks are extremely common — any oscillation will produce multiple peaks. The 3-tick minimum separation is likely too small to distinguish a meaningful double-top from normal noise. Since double-tops only tighten trailing (they don't fire entries), the failure mode is overly conservative trailing rather than bad trades — but persistent false tightening could prevent the system from riding profitable trends.

## Position sizing multiplication chain

The lot size flows through three stages in `create_signal`:

1. `position_sizer.get_lot_size()` — streak-based (1.5x after 3 wins, 0.5x after 2 losses) or Kelly (after 20 trades with auto-tune enabled)
2. `time_profiler.get_aggression_mult(hour)` — 0.5x to 2.0x based on hourly P&L history
3. `max/min` clamping to `[MIN_LOT_SIZE, MAX_LOT_SIZE]` = `[0.01, 0.05]`

The Kelly path: `kelly_fraction * BASE_LOT_SIZE / KELLY_MIN_FRACTION` = `0.25 * 0.01 / 0.01` = 0.25 lots before clamping. Clamped to 0.05 inside `get_kelly_lot_size`, then multiplied by aggression (up to 2.0x) = 0.10, then clamped again to 0.05. The double-clamping obscures the actual sizing logic and makes it hard to reason about what lot size will actually be used at any given time.
