# Fire-and-forget signals with position gating and hysteresis

The main loop no longer blocks on EA confirmation or manages its own trailing stop. Signal dispatch is fire-and-forget: write the file, update state, move on. To prevent the duplicate-position problem this creates, a `position_open` boolean gates signal firing, with a timer-based reset after `MAX_HOLD_SECONDS` (120s). A hysteresis filter adds a second gate: direction reversals require $0.50 of price movement from the last signal. The trailing stop manager (now dead code) remains in the repo but is no longer imported.

**Watch for:** Position state can desync from reality ŌĆö the timer is the only reset mechanism, meaning Python can block new signals for up to 2 minutes after a position has already been closed by the EA (confirmed). The hysteresis state persists across position boundaries, which may block valid signals after a timeout reset (likely).

## High-level view

Position gating replaces the old `trailing.is_tracking` guard with a simpler boolean, but trades one problem (duplicate positions) for another (stale gate). The EA closes positions via its own trailing stop within seconds to tens of seconds, yet Python has no way to observe that closure. The 120-second timeout is a coarse fallback that creates a dead zone where valid signals are suppressed.

Hysteresis is sound in concept ŌĆö preventing $0.10 oscillations from generating whipsaw trades ŌĆö but the state (`last_signal_direction`, `last_signal_price`) never resets on position close. After a timeout reset of `position_open`, the next signal still has to pass the hysteresis check against the price from 2 minutes ago, which in XAUUSD can be dollars away.

## Position gate without close detection

The old code used `trailing.is_tracking` as the gate ŌĆö the trailing stop manager knew when a position was open because it received the fill confirmation and tracked until exit. The new code replaces this with a boolean flipped to `True` on signal send and reset only by a wall-clock timer:

```python
def check_position_timeout():
    global position_open, position_open_time
    if position_open and (time.time() - position_open_time) >= Config.MAX_HOLD_SECONDS:
        position_open = False
```

The EA's trailing stop tiers (BE at $0.30, locks at $0.60/$1.20/$2.00) typically close positions well before 120 seconds. During that entire window, Python refuses to fire new signals. For a scalper running at 100ms loop intervals in a trending market, 2 minutes of silence is a significant opportunity cost.

The bridge already has `read_confirmation` infrastructure and the EA writes exit confirmation files. A non-blocking check of the confirm flag (already implemented in `_try_read_confirmation`) could serve as a fast reset: if the EA has written a close confirm, clear `position_open` immediately. This preserves the fire-and-forget semantics (no blocking wait) while closing the feedback gap.

## Hysteresis boundary conditions

After a position timeout reset, `last_signal_direction` still holds the direction of the previous trade. If price moved $3.00 during the 120s hold, the hysteresis check passes trivially. But if the timeout fires during a range-bound period ŌĆö say price at $2000, last signal at $2000.20, new direction opposite ŌĆö the $0.50 threshold blocks the first trade of the new context despite there being no prior position to protect.

Resetting `last_signal_direction = None` inside `check_position_timeout()` would make the first post-timeout signal always pass, matching the "first signal ever" behavior.

## Test coverage gaps

Not tested: the interaction between `check_position_timeout` and subsequent `passes_hysteresis` calls (the stale-state scenario described above), the `fire_signal` path when `bridge.write_signal` returns `False` (state should remain unchanged ŌĆö it does, but no test asserts this), and the main loop integration with both gates active simultaneously.
