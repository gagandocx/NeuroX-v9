# Phase 4 market microstructure modules

Six new trading modules (multi-timeframe confluence, liquidity sweep detection, tick frequency spike, cumulative delta, spread-as-signal, support/resistance zones) added as standalone Python files, integrated into the main trading loop, and accompanied by unit tests. The architecture follows the existing pattern: each module is a pure function taking tick/bar data and returning a signal dict, with thresholds centralized in Config. Version bumped to 9.40/20250703 across all required files.

**Watch for:** Four of six modules are computed but their outputs are silently discarded in main.py — tick frequency amplifier, MTF confluence position multiplier, and cumulative delta divergence signals never influence trading decisions (confirmed). The tick deque capacity (maxlen=50) constrains spread and delta calculations that need meaningful window sizes (confirmed). Liquidity sweep fires without any persistence or spread gating, unlike all other signals (confirmed).

## High-level view

Only three of six modules actually affect trading behavior: spread_signal gates all entries via `spread_allows_trading`, liquidity_sweep fires immediate reversal signals, and support_resistance informs mean-reversion direction in ranging mode. The remaining three (tick_frequency, mtf_confluence, cumulative_delta) are called but their results are assigned to local variables that are never read again.

TickCollector's `_tick_prices` and `_tick_timestamps` deques have a maxlen of 50. During institutional spikes (50+ tps), 50 ticks covers only 1 second — making the cumulative delta's 30-tick window represent 0.6 seconds of market data, too thin for meaningful divergence detection precisely when tick_frequency signals that institutional activity is present.

Integration-level tests covering the main loop's use of these modules (spread gating, sweep firing, SR bounce feeding mean reversion) do not exist.

## Tick frequency, MTF confluence, and delta: computed but discarded

In the main loop, three modules produce outputs that are never consumed:

```python
tick_freq_info = compute_tick_frequency(tick_collector._tick_timestamps)
# tick_freq_info never referenced again

mtf_info = compute_mtf_confluence(tick_collector._completed_bars)
# mtf_info never referenced again

delta_info = compute_cumulative_delta(tick_collector._tick_prices)
divergence_info = detect_delta_divergence(tick_collector._tick_prices)
# Neither delta_info nor divergence_info referenced again
```

The spec calls for tick frequency to amplify conviction when momentum aligns, MTF confluence to scale position size (1.5x when all timeframes agree, 0.5x when M1 disagrees), and delta divergence to provide early reversal warnings. None of this wiring exists. The modules themselves are correctly implemented — the gap is purely in integration.

## Liquidity sweep fires without full gating

The sweep signal path in main.py:

```python
if sweep_info["detected"] and can_trade() and direction_matches_open(sweep_info["direction"]):
    fire_signal(bridge, sweep_info["direction"], current_price,
                "LIQUIDITY SWEEP", regime="momentum", model_name="sweep_v9")
```

This bypasses `spread_allows_trading`, `passes_atr_filter`, `passes_tick_consistency`, and `passes_signal_persistence`. The existing velocity spike path also bypasses persistence, so there's precedent for instant signals skipping some filters. However, the spread gate is specifically designed to block entries during news events (spread spike + fast move), and a liquidity sweep — which is a fast price spike — is exactly the scenario where spread tends to widen. Firing during a spread spike could enter at the widest cost.

## Spread-as-signal: proxy measures volatility, not actual spread

The spread module approximates spread from median absolute tick-to-tick differences because the tick file only provides bid prices. This proxy measures price volatility, not the actual bid-ask spread. A fast trending move produces large tick-to-tick diffs (proxy says "wide spread") even though actual spread may be normal. The module will suppress trading during fast directional moves in trending regimes, which conflicts with the system's core purpose of capturing momentum.

## Support/resistance: breakout signals unused

The S/R bounce signals feed the mean-reversion strategy in ranging mode:

```python
if mean_rev_direction == "FLAT" and sr_info["signal_type"] in ("BOUNCE_UP", "BOUNCE_DOWN"):
    if sr_info["signal_type"] == "BOUNCE_UP":
        mean_rev_direction = "BUY"
```

The breakout signals (`BREAKOUT_UP`, `BREAKOUT_DOWN`) from S/R are computed but never consumed. These could inform the trending regime — a breakout through a strong level with high velocity is a high-conviction momentum entry — but currently only bounce signals influence decisions.

## Tick deque capacity under institutional activity

TickCollector's deques are capped at maxlen=50. At 10-15 ticks/second (normal), 50 ticks covers 3-5 seconds. At 50+ tps (institutional spike), 50 ticks covers only 1 second. The cumulative delta's 30-tick window at 50 tps represents 0.6 seconds of market data — insufficient for detecting meaningful price-vs-delta divergence. If tick_frequency detection were wired in to trigger delta analysis during spikes, the delta window would be too short precisely when it matters most.

## Test coverage gaps

Not tested: the main.py integration paths — spread gating blocking a signal, sweep detection firing through main's filters, S/R bounce overriding mean reversion direction. Not tested: behavior when tick deque is at capacity during high-frequency scenarios. Not tested: interaction between multiple modules (e.g., sweep firing while spread is in SPIKE state).
