# Full standalone rewrite of NeuroX EA from Python+bridge to single MQL5 file

The entire Python-derived trading strategy (50 EMA trend filter, 9 EMA entry range, 5-indicator choppy filter, EMA-based SL, candle-close exit, $5 breakeven, reversal detection) has been collapsed into a single 1347-line MQL5 Expert Advisor. The Python system's real-time bridge architecture (file I/O between Python and MT5) is eliminated; all logic now runs natively inside MetaTrader. The approach is a faithful line-by-line translation of each Python module's behavior into MQL5 equivalents with native indicator handles.

**Watch for:** SL placement uses bid as reference for BUY orders, making the effective SL tighter by one spread compared to the Python system that measured from entry (ask) [confirmed]. The candle-close exit PnL check uses raw price math rather than the position's broker-reported profit, creating a divergence when swap/commission is material [confirmed]. Average ATR window is 3x wider than the Python system's, changing when the ATR ratio voter fires [confirmed].

## High-level view

The trend direction logic mirrors the Python: price must be above both 50 EMA and 9 EMA for BUY (below both for SELL), with a "conflicting" state when price is between them that blocks all entries.

The choppy market filter translates all five voters (ADX, Choppiness Index, BB Width squeeze, ATR ratio, Variance Ratio) into MQL5. The Choppiness Index is computed manually from ATR sums and highest-high/lowest-low. The Variance Ratio faithfully replicates the Python's numpy-based computation. However, the average ATR used for the ATR-ratio voter is computed over 42 bars vs the Python's 14, making this voter less responsive to short-term volatility drops.

The SL placement logic has a spread-related asymmetry: for BUY entries the EA computes SL distance from bid then places SL at `bid - distance`, but the actual entry fills at ask. The Python system applied the distance from the entry price (ask). This means the standalone SL is tighter by one spread for BUY trades.

The ticket tracking mechanism (`g_trackedTicket`) relies on `ResultDeal()` or `ResultOrder()` matching what appears in the position list. If the broker uses a different ticket for the position than for the deal/order, the EA will lose track of the position on the next tick and falsely declare an SL hit.

## SL placement asymmetry relative to entry price

In `TryEntry`, the BUY path:

```mql5
double slDistance = MathAbs(price - ema60Val);  // price = bid parameter
slPrice = NormalizeDouble(bid - slDistance, digits);
```

The trade executes at ask via `g_trade.Buy(InpLotSize, _Symbol, ask, slPrice, ...)`. So the effective SL distance from entry is `ask - (bid - slDistance)` = `spread + slDistance`. The Python system computed `sl_distance = abs(current_price - ea_ema_sl)` where `current_price` was bid, then the bridge EA placed SL at `entry_price - sl_distance`. Since `entry_price` = ask, the Python SL was at `ask - |bid - ema60|`. The MQL5 SL is at `bid - |bid - ema60|`, one spread tighter.

For SELL trades: MQL5 places SL at `ask + slDistance` but entry is at bid, making the effective distance `(ask + slDistance) - bid` = `spread + slDistance` — wider than the Python equivalent by one spread.

Fix: compute `slDistance` from the execution price (`ask` for BUY, `bid` for SELL) rather than the `price` parameter which is always bid.

## Average ATR computation window

The Python system:
```python
computed_atr, computed_avg_atr = compute_atr(bars_df, period=Config.ATR_RATIO_PERIOD)
```

This computes True Range over 14 bars and returns both the current ATR and the average over those 14 bars. The MQL5 `ComputeAvgATR` takes `ATR_PERIOD * 3 = 42` indicator ATR values and averages them. With 42 bars of averaging, the MQL5 average will be smoother and slower to respond to volatility shifts, meaning the ATR ratio voter will trigger less frequently — making the choppy filter more permissive than the Python system.

## Candle-close exit — profit gate

```mql5
double priceDiff = isBuy ? (price - entryPrice) : (entryPrice - price);
double actualPnl = priceDiff * InpLotSize * InpContractSize;
if(actualPnl > 0.0)
```

This ignores swap and commission. A position that moved $0.02 in favor over several hours could be net-negative once swap is applied. The standalone EA has access to `g_position.Profit() + Swap() + Commission()` — the data is already fetched on the next line for the log message. Using broker-reported PnL would be both simpler and more accurate.

## Entry ordering race with `g_trackedTicket`

The MQL5 entry flow sets `g_trackedTicket = g_trade.ResultDeal()` immediately after a successful trade. If `ResultDeal()` returns 0 (which happens with some brokers where deal tickets are delayed), it falls back to `g_trade.ResultOrder()`. The position management loop then scans for `g_position.Ticket() == g_trackedTicket`. If the order ticket differs from the position ticket (possible with some brokers), the position won't be found.

`DetectSLHit` scans all open positions for the tracked ticket. If the ticket mismatch occurs, it declares the position gone on the very next tick, resets tracking, and allows a new entry — while the old position remains open and unmanaged (no breakeven, no candle-close exit, no reversal detection). The only protection remaining is the emergency loss limit.

A safer approach: after entry, wait one tick, then scan positions by magic number + symbol to find the newly opened position and store its actual `Ticket()`.
