# Intelligence dashboard panel via shared file bridge

Python now writes a pipe-delimited intelligence file every loop iteration containing regime, ATR status, tick flow, signal persistence, active strategy, and an overall decision (TRADING / FILTERED / WAITING). The EA reads this file on timer and renders a color-coded INTELLIGENCE section on the dashboard. The implementation restructures the main loop's filter chain from short-circuit `and` to eager evaluation so the blocking filter can be identified and reported.

**Watch for:** The removal of short-circuit evaluation means `passes_signal_persistence()` now runs every non-FLAT iteration regardless of whether earlier filters already blocked — this inflates the persistence counter faster than before (confirmed). Also, in the trending bar-based path, `passes_atr_filter(atr)` is called twice per cycle — once for intelligence reporting and again inside the filter block (confirmed, harmless but wasteful).

## High-level view

The core design change is not the file I/O or the dashboard rendering — it's the restructuring of the filter chain. Previously, filters short-circuited: if cooldown failed, nothing downstream ran. Now all filters execute unconditionally so that the first failing filter can be named in the intelligence file. This is necessary for the "why isn't it trading" requirement but changes the semantics of `passes_signal_persistence`, the only stateful filter.

The file protocol is a single pipe-delimited line (10 fields), overwritten atomically via `Path.write_text`. The EA reads it with `FILE_COMMON` on every `OnTimer` tick. No versioning, no timestamps, no staleness detection — the EA always displays whatever it last read, meaning the display can show stale intelligence indefinitely if the Python process stalls.

## Persistence counter semantics changed by eager evaluation

The old code:

```python
if (tick_momentum != "FLAT"
        and can_trade()
        and has_capacity(tick_strength)
        and passes_hysteresis(tick_momentum, current_price)
        and passes_tick_consistency(tick_collector)
        and passes_signal_persistence(tick_momentum)):
```

`passes_signal_persistence` increments (or resets) the global `signal_persistence_count` as a side effect. Under short-circuit evaluation it only executed when all prior filters passed — meaning the counter only advanced when the system was "almost ready to trade." The new code calls it unconditionally whenever momentum is non-FLAT:

```python
filter_persistence = passes_signal_persistence(tick_momentum)
```

During cooldown (e.g., 30 seconds after last trade), the persistence counter still counts up. When cooldown expires, persistence may already be satisfied, enabling an immediate trade that previously would have needed N more cycles to build up. The practical impact depends on `SIGNAL_PERSISTENCE_COUNT` and `COOLDOWN_SECONDS` — if persistence requires only 2-3 signals and the loop runs every second, the counter reaches threshold well within any cooldown window.

The same change applies to all three paths (tick-based, trending bar-based, ranging bar-based), representing a behavioral loosening across the board.

## Double ATR filter call

In the bar-based trending path:

```python
intel_atr_pass = passes_atr_filter(atr)   # for intelligence reporting
...
filter_atr = passes_atr_filter(atr)        # for trade gating
```

The function is pure, so this is functionally harmless. But the debug logger fires twice per cycle when ATR is low. Reusing `intel_atr_pass` for the filter check eliminates the duplicate log and the redundant call.

## EA reads with no freshness signal

`ReadIntelligenceFile()` opens the file, parses it, and closes — no timestamp comparison against `TimeCurrent()`. If the Python bridge dies, the dashboard will show the last intelligence state (potentially "TRADING") indefinitely. The heartbeat age mechanism exists for liveness detection, but the intelligence display itself gives no visual indication of staleness. A simple mitigation: if heartbeat age exceeds a threshold (already computed elsewhere in the EA), override the intelligence display with a "STALE" indicator.

## Test coverage

Not tested: the intelligence-state computation logic in `main.py` — which filter gets reported when multiple fail, the decision/reason mapping, the tick-based vs bar-based intelligence output differences. This logic is embedded in the main loop and not extracted into a testable function. The persistence-counter behavioral change described above is also untested — no test verifies that persistence still gates trades correctly under the new eager-evaluation model.
